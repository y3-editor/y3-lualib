# 策划案生成 — 多轮引导详细指南

> 本文件是 `y3-game-spec` 阶段 1 的详细执行指南。
> 被 `SKILL.md` 引用，激活阶段 1 时必须读取本文件。

---

## 🔄 引导流程总览

```
Step 1 游戏类型       ← 确定品类和核心定位
    │
Step 2 核心玩法       ← 明确玩什么、怎么玩
    │
Step 3 角色与单位     ← 玩家控制谁、对抗谁
    │
Step 4 战斗与技能     ← 战斗方式、技能设计
    │
Step 5 经济与成长     ← 资源获取/消耗、升级体系
    │
Step 6 UI 与交互      ← 需要哪些界面、操作方式
    │
Step 7 胜负与流程     ← 怎么赢/怎么输、关卡/波次流程
    │
Step 8 数值与平衡     ← 关键数值框架
    │
Step 9 可行性收束审查  ← ⭐ 对照 Y3 引擎能力边界，驳回/降级不合理需求（强制）
    │
    ▼
汇总生成策划案 → 用户审阅确认 → 进入阶段 2
```

> ⚠️ **每一步都必须使用 `ask_user_question` 与用户交互**，等用户回答后再进入下一步。
> 用户可随时说"跳过"来使用默认值，AI 会基于前面的回答自动补全。

## Step 1：游戏类型与核心定位

**目的**：确定游戏品类，后续所有引导内容会基于此调整。

**使用 `ask_user_question` 提问**：

```
问题: 你想做什么类型的游戏？
选项:
  - 塔防 — 建造防御塔阻挡敌人波次
  - RPG — 角色扮演，成长+战斗+剧情
  - 肉鸽/Roguelike — 随机关卡+永久死亡+构建流派
  - 生存 — 收集资源、抵御威胁、活下去
  - MOBA — 多人对战，推塔/团战
  - 竞技/休闲 — 小游戏、派对游戏
  - [其他] （用户自定义输入）
```

**AI 后续动作**：
- 记录 `game_type`
- 根据类型加载对应的引导模板（见「游戏类型引导差异表」）

### 游戏类型引导差异表

| 类型 | Step 3 侧重 | Step 4 侧重 | Step 5 侧重 | Step 7 侧重 |
|------|-------------|-------------|-------------|-------------|
| **塔防** | 塔种类+敌人种类 | 塔攻击方式+特殊效果 | 金币经济 | 波次流程+基地血量 |
| **RPG** | 英雄职业+怪物层级 | 技能树+Buff+装备 | 经验+金币+掉落 | 关卡/Boss+主线进度 |
| **肉鸽** | 初始角色+遭遇物 | 技能/道具随机池 | 局内资源+局间传承 | 层数+Boss+死亡重来 |
| **生存** | 玩家角色+威胁源 | 采集/建造/战斗 | 资源种类+消耗 | 存活时长/目标达成 |
| **MOBA** | 英雄池+兵线+野怪 | 4技能+被动+装备 | 金币+经验+装备商店 | 推塔+水晶+团战 |

## Step 2：核心玩法描述

**目的**：让用户用自己的话描述游戏的核心循环。

**使用 `ask_user_question` 提问**：

```
问题: 用一两句话描述你的游戏核心玩法——玩家主要在做什么？
（AI 根据 Step 1 的类型给出参考示例）

选项（根据 game_type 动态生成）:
  例如塔防类:
  - 在固定路线上建造防御塔，抵挡越来越强的怪物波次
  - 自由摆放防御塔，怪物从多个方向进攻
  - 英雄+塔混合，英雄可以移动战斗
  例如 RPG 类:
  - 选择英雄，击败波次怪物，收集装备变强
  - 探索地图，完成任务，击败 Boss
  - 多英雄组队，技能配合，刷副本
  - [其他] （用户自定义描述）
```

**AI 后续动作**：
- 记录 `core_gameplay`
- 提炼出 1 句「核心循环」描述，后续写入策划案的"游戏概述"

## Step 3：角色与单位设计

**目的**：明确游戏中有哪些可控/不可控单位。

**使用 `ask_user_question` 提问**（可多选）：

```
问题: 游戏中需要哪些角色/单位？（可多选）

选项（根据 game_type 动态调整）:
  例如通用:
  - 玩家英雄（可选择不同职业/角色）
  - 近战小怪
  - 远程小怪
  - Boss
  - NPC（友好单位、商店等）
  - 建筑/塔（可放置的防御建筑）
  - 召唤物（技能召唤的临时单位）
  - [其他] （用户自定义）
```

**追问**（如果选了"玩家英雄"）：

```
问题: 需要几个可选英雄？每个英雄有什么特色？
选项:
  - 3 个英雄（战士/法师/射手，经典三角）
  - 5 个英雄（更丰富的选择）
  - 1 个固定英雄（不需要选择）
  - [其他] （用户自定义数量和描述）
```

**AI 后续动作**：
- 记录 `units` 列表（含玩家单位、敌方单位、NPC）

## Step 4：战斗与技能系统

**目的**：确定战斗方式和技能设计方向。

**使用 `ask_user_question` 提问**：

```
问题: 战斗系统怎么设计？
选项（根据 game_type 动态调整）:
  - 自动攻击+主动技能（类 RTS/MOBA，选中单位自动打，技能手动放）
  - 纯自动战斗（放置类，玩家只负责阵容/建造）
  - 回合制（你打一下我打一下）
  - 键盘操作（WASD 移动 + 技能键）
  - [其他]
```

**追问**（技能数量）：

```
问题: 每个英雄/塔需要几个主动技能？
选项:
  - 1 个（简单明了）
  - 2-3 个（适中复杂度）（Recommended）
  - 4 个（MOBA 风格，QWER）
  - 不需要主动技能（纯自动/纯被动）
```

**追问**（Buff/特殊效果）：

```
问题: 需要哪些特殊效果/Buff？（可多选）
选项:
  - 减速
  - 中毒/持续伤害
  - 眩晕/定身
  - 护盾/加血
  - 增伤/增速
  - 不需要特殊效果
  - [其他]
```

**AI 后续动作**：
- 记录 `combat_style`、`skill_count`、`buffs`
- 结果将写入"战斗系统"和"技能设计"章节

```
  - 物品栏/背包（显示持有物品）
```

**追问**（操作方式）：

## Step 7：胜负条件与游戏流程

**目的**：明确游戏的开始、进行、结束规则。

**使用 `ask_user_question` 提问**：

```
问题: 游戏怎么算赢？怎么算输？
选项（根据 game_type 动态调整）:
  例如通用:
  - 打完所有波次获胜，基地被摧毁失败
  - 击败最终 Boss 获胜，英雄死亡且无复活次数失败
  - 存活指定时间获胜，血量归零失败
  - 推掉敌方水晶获胜，己方水晶被推失败
  - [其他]
```

**追问**（游戏流程）：

```
问题: 游戏的流程是怎样的？
选项:
  - 波次制（第1波→第2波→...→最终波）
  - 自由探索（开放地图，自由进行）
  - 关卡制（第1关→第2关→...）
  - 限时挑战（倒计时内完成目标）
  - [其他]
```

**追问**（如果是波次制）：

```
问题: 大约需要多少波？
选项:
  - 5 波（短局，适合快节奏）
  - 10 波（中等长度）（Recommended）
  - 15-20 波（长局，有更多策略深度）
  - [其他]
```

**追问**（死亡与复活）：

```
问题: 英雄死亡后怎么处理？
选项:
  - 自动复活（N 秒后在出生点复活）（Recommended）
  - 消耗资源复活（花金币/复活次数）
  - 永久死亡（死了就结束）
  - 不存在死亡机制
  - [其他]
```

**AI 后续动作**：
- 记录 `win_condition`、`lose_condition`、`game_flow`、`death_mechanic`
- 结果将写入"胜负条件"和"游戏流程"章节

## Step 9：可行性收束审查（强制，不可跳过）

**目的**：在生成最终策划案前，对照 Y3 引擎能力边界逐项审查，驳回/降级不合理需求。

> ⚠️ **此步骤为强制步骤**，即使用户说"跳过"也不可跳过。
> 所有 ❌ 不可行项必须处理后才能进入汇总环节。

### 执行步骤

**Step 9.1: 逐项扫描**

遍历 Step 1~8 收集到的所有功能点，逐一对照 SKILL.md 中的「Y3 引擎能力边界（红线清单）」，分为三类：

| 分类 | 含义 | 处理方式 |
|------|------|---------|
| ✅ 可行 | Y3 原生支持或有成熟方案 | 无需处理，直接进入策划案 |
| ⚠️ 需变通 | 可做但需降级/简化 | 向用户说明差异，协商后写入 |
| ❌ 不可行 | 触碰引擎红线 | 必须驳回，提供替代方案 |

**Step 9.2: 驳回/协商**

对于 ❌ 项，**使用 `ask_user_question` 提问**：

```
问题: 以下需求超出 Y3 引擎能力，需要调整：
  ❌ [具体功能] — 原因：[为什么不行]
  建议替代方案：[替代方案描述]

选项:
  - 接受替代方案
  - 去掉该功能
  - [其他]（用户自定义调整）
```

对于 ⚠️ 项，**使用 `ask_user_question` 提问**：

```
问题: 以下需求需要做一些调整才能在 Y3 中实现：
  ⚠️ [具体功能] — 原始需求 vs 变通方案的差异说明

选项:
  - 接受变通方案
  - 保留原始需求（标注风险）
  - [其他]
```

**Step 9.3: 更新功能收集结果**

将收束后的最终决策记录下来，用于更新策划案第 10 章「技术可行性说明」：
- ❌ 项记录驳回原因 + 最终采纳的替代方案
- ⚠️ 项记录原始需求 + 采纳的变通方案
- ✅ 项直接标注"原生支持"

### AI 后续动作
- 收束结果将写入策划案第 10 章
- 所有 ❌ 项处理完毕后，进入「汇总生成策划案」环节

## 用户审阅确认

策划案生成后，**必须使用 `ask_user_question` 请用户确认**：

```
问题: 策划案已生成，请审阅。是否满意？
选项:
  - ✅ 满意，进入阶段 2（开始拆分执行案）
```

如果用户选择修改，进入**局部修改模式**：
- 用户指出需要修改的章节
- AI 更新对应章节
- 再次请用户确认
- 最多循环 3 轮，之后强制进入阶段 2

## ⚠️ 阶段 1 关键规则

| 规则 | 说明 |
|------|------|
