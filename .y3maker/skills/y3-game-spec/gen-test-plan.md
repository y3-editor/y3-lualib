# 验收标准自动生成规范

> 当测试案不存在时，AI 按本文档流程自动生成。
> 文件路径和命名由 `.codemaker/spec-config.json` 配置决定（默认 `openspec/docs/{gameName}测试案.md`）。
> Fallback：`openspec/docs/AutoTestPlan.md`（旧命名兼容）。
> 当测试案已存在时，**必须先执行 `doc-consistency.mdc` 规则 4 的一致性校验**，确认测试案与当前策划案匹配后才能使用。

```
策划案文档

    ▼
┌─────────────────────────────────────────────────────────────────────┐
│ Step A: 解析策划案                                                    │

│ 2. 提取开发阶段列表（Phase 1, 2, 3 ...）                            │
│ 3. 提取每个阶段的功能清单和验收点                                     │
│ 4. 提取使用的技能（y3-obj-gen / y3-ui-pipeline / y3-lua-pipeline）   │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│ Step B: 映射测试用例                                                  │
│                                                                     │
│ 对策划案中的每个验收点，按"验收点→测试用例映射规则"生成测试用例         │
│ 每个用例必须包含：编号、测试项、执行方法、判定条件                     │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│ Step C: 填充模板                                                      │
│                                                                     │
│ 按"文档结构模板"组装完整的 AutoTestPlan.md                           │
│ 包含：整体流程图、Step 0 清理、各 Phase、Step 5 回归、Step Final 报告 │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│ Step D: 自检                                                          │
│                                                                     │
│ 执行生成质量检查清单（见下方），全部通过后输出文件                     │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
  输出: {paths.testPlan}/{gameName}测试案.md
  （路径和文件名由 .codemaker/spec-config.json 配置）
```

## 📌 Step A: 策划案解析规则

从策划案中必须提取以下信息（缺失项标记 ⚠️ 并用合理默认值填充）：

| 提取项 | 来源 | 缺失时默认值 |
|--------|------|-------------|
| **游戏概述** | "游戏概述" 章节 | 用策划案标题概括 |
| **开发阶段** | "开发阶段总览" 或各 Phase 标题 | 按功能分类自动划分 |
| **功能清单** | 每个 Phase 的功能列表 | 从验收点反推 |
| **验收点** | "详细验收点" 或各功能的验收描述 | 从功能清单推导 |
| **使用技能** | Phase 对应的 skill 标注 | 根据功能类型推断 |
| **核心规则** | "核心规则" 章节 | 从验收点提取约束 |
| **UI 布局** | UI 相关章节的布局描述 | 仅列出画板名称 |
| **数值规则** | 波次/金币/经验等具体数值 | 标记 ⚠️ 需确认 |

## 📌 Step B: 验收点→测试用例映射规则

### 通用映射表

| 验收点类型 | 测试方法 | 使用工具 | 判定条件模式 |
|------------|----------|----------|-------------|
| 物编可创建 | execute_lua 创建实体 | `execute_lua` | 🤖 创建成功=PASS |
| 物编字段正确 | execute_lua 读取属性 | `execute_lua` | 🤖 属性值符合预期=PASS |
| UI 画板存在 | 检查控件树 | `get_ui_canvas` | 🤖 画板存在=PASS |
| UI 控件完整 | 深度检查控件 | `get_ui_canvas depth:-1` | 🤖 控件齐全=PASS |
| UI 视觉效果 | 截图存证 | `capture_screenshot` | 📸 截图存证 |
| 按钮点击交互 | 点击并验证结果 | `bot_click` + `get_logs` | 🤖 日志/状态变化=PASS |
| 键盘快捷键 | 按键并验证 | `bot_hotkey` + `get_logs` | 🤖 日志/状态变化=PASS |
| 数值变化 | 前后对比 | `execute_lua`×2（前/后读取） | 🤖 数值增减符合预期=PASS |
| 状态切换 | 触发+检查 | `execute_lua` 触发 + 检查 | 🤖 状态已切换=PASS |
| UI 数据绑定 | Lua 值与 UI 对比 | `execute_lua` + `get_ui_canvas` | 🤖 数值一致=PASS |
| 定时器/延迟 | 等待+验证 | `execute_lua` + 等待 N 秒 | 🤖 定时触发=PASS |
| 游戏启动无 Trace | 启动+读日志 | `launch_game` + `get_logs` | 🤖 无 Trace=PASS |
| Lua 语法检查 | 静态分析 | `read_problems_lua` | 🤖 无错误=PASS |
| 弹窗/对话框 | 点击+检查 | `bot_click` + `get_ui_canvas` | 🤖 弹窗出现/消失=PASS |

### 编号规则

```
T-{序号}      主测试用例，如 T-01, T-02, T-27
T-{序号}{字母}  子测试用例，如 T-27a, T-27b, T-27c
```

- 按 Phase 分段，每个 Phase 内连续编号
- 每个 Phase 必须以"Lua 语法检查"和"游戏启动无 Trace"开头（冒烟测试）
- 子用例用于拆分复杂验收点（如技能系统拆分为挂载/释放/伤害/消耗/冷却等）

## 📌 Step D: 生成质量自检清单

生成完成后，AI 必须逐项检查以下内容：

```
□ 1. 覆盖率：策划案每个验收点都有对应的测试用例（1:1 或 1:N）
□ 2. 可自动化：每个用例都标注了执行工具（无"人工检查"/"目视"等）
□ 3. 判定明确：每个用例都有 PASS/FAIL 判定条件
□ 4. 编号连续：T-01 开始，无跳号，子用例字母连续
□ 5. 冒烟齐全：每个 Phase 以 Lua 语法检查 + 启动无 Trace 开头
□ 6. Step 0 存在：清理步骤完整
□ 7. Step 5 存在：修复回归流程完整
□ 8. Step Final 存在：报告模板完整
□ 9. 日志规范：定义了 [UI-LOG] 格式
□ 10. 工具匹配：所有用例使用的工具都在可用工具列表内
     可用工具：execute_lua / bot_click / bot_hotkey / get_logs /
              get_ui_canvas / get_ui_center_in_windows /
              capture_screenshot / read_problems_lua /
              launch_game / quick_restart / get_game_status
□ 11. 测试护盾：如涉及战斗/死亡机制，包含护盾注入章节
□ 12. 前置条件：需要日志验证的用例（如技能释放），包含前置条件检查清单
```

自检不通过时，AI 必须自行修正后再输出文件。
