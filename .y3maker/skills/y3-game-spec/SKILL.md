# Y3 游戏开发全流程编排器

管理从零开始制作游戏的完整闭环：**策划案 → 执行案 → 制作 → 测试案 → 测试 → 迭代**。

## � 配置文件（spec-config.json）

> **路径**：`.codemaker/spec-config.json`
> 用户可通过此文件自定义四类文档的输出路径和 HTML 报告设置。

### 何时读取配置

| 时机 | 操作 |
|------|------|
| 阶段 1 策划案确认后 | 将游戏名称写入 `gameName` 字段 |
| 每次生成/读取文档时 | 读取配置，解析文件路径和文件名 |
| 测试报告生成后 | 读取 `htmlReport.autoOpen` 决定是否自动打开 |
| 测试报告生成后 | 读取 `svn.autoUpload` 决定是否自动提交到 SVN |

### 文件命名规则（强制）

所有流程文档必须按以下规则命名，**禁止使用旧的硬编码文件名**：

| 文档类型 | 文件名模板 | 示例 |
|----------|-----------|------|
| 策划案 | `{gameName}设计案.md` | `塔防设计案.md` |
| 执行案 | `{gameName}执行案.md` | `塔防执行案.md` |
| 测试案 | `{gameName}测试案.md` | `塔防测试案.md` |
| 测试报告 | `{gameName}测试报告-{timestamp}.md` | `塔防测试报告-2026年4月14日11时12分54秒.md` |
| HTML报告 | `{gameName}测试报告-{timestamp}.html` | `塔防测试报告-2026年4月14日11时12分54秒.html` |

**占位符说明**：
- `{gameName}`：从 `spec-config.json` 的 `gameName` 字段获取
- `{timestamp}`：生成时刻的时间戳，格式为 `YYYY年M月D日HH时MM分SS秒`

### 路径解析流程

```
1. 读取 .codemaker/spec-config.json
2. 获取 paths.{docType} → 输出目录
3. 获取 fileNames.{docType} → 文件名模板
4. 替换 {gameName} 和 {timestamp} 占位符
5. 拼接为完整路径 → 输出文件
```

### 配置为空时的处理（强制）

进入任何阶段前，读取 `spec-config.json` 后检查关键字段：

```
读取 spec-config.json
    │
    ▼
gameName 为空？
    ├─ 是 → ask_user_question 询问游戏/项目名称
    │       → 用户回答后写入 gameName 字段
    └─ 否 → 继续

paths 中任一值为空？
    ├─ 是 → ask_user_question 询问用户期望的输出路径
    │       → 用户回答后写入对应字段（可选择使用默认值）
    └─ 否 → 继续

svn.autoUpload = true 但未安装 svn？
    ├─ 是 → 提示用户：SVN 未安装，已自动关闭上传功能
    └─ 否 → 继续
    │
    ▼
✅ 配置完整，进入流程
```

**规则**：
- `gameName` 为空时**必须**通过 `ask_user_question` 向用户询问，不得自行编造
- 如果用户在阶段 1 策划案中已明确游戏名称，可直接提取并**告知用户已自动填写**
- 询问示例：`"检测到配置文件中游戏名称为空，请输入你的游戏/项目名称（如：塔防、肉鸽RPG）："`

### 旧文件名兼容

如果项目中已存在旧命名的文件（如 `GameDesign.md`、`Y3AgentTestPlan.md`），按以下规则处理：
1. 读取时：优先查找新命名，找不到再 fallback 到旧命名
2. 写入时：始终使用新命名规则
3. 不主动重命名已有文件，避免破坏引用

## 📋 阶段 1：生成策划案（多轮引导 + 可行性收束）

### 触发条件
用户说"做一个游戏"、"做个塔防"等。

### 输出物
按 `spec-config.json` 配置输出：`{paths.designDoc}/{gameName}设计案.md`

> 策划案确认后，自动将游戏名称写入 `spec-config.json` 的 `gameName` 字段。

### 执行指南
> 📘 **详细引导流程**：`skills/y3-game-spec/game-design-guide.md`
> 进入阶段 1 时**必须读取**该文件，按其中的 Step 1-8 逐步引导用户。

### 流程概要

```
Step 1 游戏类型 → Step 2 核心玩法 → Step 3 角色单位 → Step 4 战斗技能
    → Step 5 经济成长 → Step 6 UI交互 → Step 7 胜负流程 → Step 8 数值平衡
    → ⭐ Step 9 可行性收束审查（新增）
    → 汇总生成策划案 → 用户审阅确认 → 进入阶段 2
```

### 🔒 可行性收束机制（强制，Step 9）

> **目的**：在策划案最终确认前，对照 Y3 引擎能力边界进行收束审查，
> 驳回/降级明显不合理的需求，保证策划案中的每一项功能都可以在 Y3 中实现。

#### Y3 引擎能力边界（红线清单）

| ❌ 不可能实现 | 原因 | 建议替代方案 |
|---------------|------|-------------|
| 开放世界/大世界 | Y3 为固定地图引擎，无无缝加载/流式地图 | 分区地图 + 传送门 |
| 3D 第一/第三人称射击 | 无 FPS/TPS 相机和射线检测 | RTS 视角 + 自动攻击 |
| 物理驱动玩法（弹球/布料/车辆） | 无通用物理引擎 | 用动画 + 逻辑模拟 |
| 大规模多人在线（MMO 100+人） | 房间制，通常 ≤12 玩家 | 12 人以内对战/合作 |
| 实时语音/视频 | 引擎不提供 | 文字聊天 / 快捷信号 |
| 自定义着色器/后处理 | 无 Shader 编辑能力 | 使用引擎预设特效 |
| 过场动画/CG | 无电影级动画系统 | 镜头移动 + UI 叙事 |
| 复杂 AI 寻路（3D 导航网格） | 仅支持 2D 寻路 | 简单路径点 + 区域控制 |
| 持久化存档跨局继承 | 局内运行环境，局间数据有限 | 使用存档 API 存关键数据 |
| 超复杂 UI（拖拽排序/富文本编辑） | UI 系统能力有限 | 简化交互设计 |

#### 收束审查流程（Step 9 执行步骤）

```
Step 8 完成后
    │
    ▼
┌─ Step 9.1: 逐项扫描 ─────────────────────────────────────────────────┐
│ 遍历 Step 1~8 收集到的所有功能点，逐一对照「红线清单」                    │
│ 输出分类结果：                                                         │
│   ✅ 可行      — Y3 原生支持或有成熟变通方案                            │
│   ⚠️ 需变通    — 可做但需降级/简化，标注变通方案                        │
│   ❌ 不可行    — 触碰红线，必须驳回                                     │
└────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 9.2: 不可行项驳回 + 变通项协商 ────────────────────────────────┐
│ 对于 ❌ 不可行项：                                                     │
│   - 向用户说明原因                                                     │
│   - 提供替代方案建议                                                   │
│   - 使用 ask_user_question 让用户选择：接受替代 / 去掉该功能          │
│                                                                        │
│ 对于 ⚠️ 需变通项：                                                    │
│   - 说明原始需求与变通方案的差异                                       │
│   - 使用 ask_user_question 让用户确认变通是否接受                     │
└────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 9.3: 更新策划案 ───────────────────────────────────────────────┐
│ 将收束结果写入策划案第 10 章「技术可行性说明」                           │
│ 驳回项标注 ❌ + 替代方案                                               │
│ 变通项标注 ⚠️ + 最终采纳方案                                          │
│ 可行项标注 ✅                                                          │
└────────────────────────────────────────────────────────────────────────┘
    │
    ▼
汇总生成策划案 → 用户审阅确认 → 进入阶段 2
```

#### 常见不合理需求示例及应对

| 用户需求 | 判定 | AI 应对 |
|----------|------|---------|
| "做个开放世界 RPG" | ❌ | "Y3 不支持开放世界，建议改为固定地图 + 多区域传送的 RPG" |
| "做个 FPS 射击游戏" | ❌ | "Y3 没有 FPS 视角，建议改为 RTS 视角的射击塔防" |
| "支持 100 人大乱斗" | ❌ | "Y3 最多 12 玩家，建议改为 6v6 或 FFA 12 人" |
| "做个有物理碰撞的弹球" | ❌ | "Y3 没有物理引擎，建议改为回合制弹射（逻辑模拟）" |
| "要有过场 CG 动画" | ⚠️ | "建议用镜头移动 + UI 文本叙事代替" |
| "做个 50 波的塔防" | ⚠️ | "波次过多可能导致一局过长，建议 10~20 波并增加难度曲线" |
| "英雄要有 10 个技能" | ⚠️ | "技能过多操作复杂，建议 3~4 个主动 + 1 被动，其余通过装备/Buff 实现" |

### 关键规则

| 规则 | 说明 |
|------|------|
| 每步必须 `ask_user_question` | 禁止跳过任何一步的用户交互 |
| 选项动态适配 | 根据 Step 1 的游戏类型调整后续选项 |
| 用户可跳过 | AI 基于已有信息自动补全 |
| **收束审查不可跳过** | Step 9 为强制步骤，所有 ❌ 项必须处理后才能生成最终策划案 |
| 确认才进入阶段 2 | 策划案必须经用户明确确认 |

### 策划案包含章节（10 章）

1. 游戏概述 → 2. 核心规则 → 3. 单位清单 → 4. 战斗系统 → 5. 经济与成长
→ 6. UI 布局设计 → 7. 游戏流程 → 8. 数值设计 → 9. 功能模块优先级 → 10. 技术可行性

## 📋 阶段 2：拆分为执行案 + API 预检 + 制作

### 触发条件
策划案已确认，用户说"开始制作"或"拆分为执行案"。

### 输入
策划案文档

### 输出物
按 `spec-config.json` 配置输出：`{paths.executionDoc}/{gameName}执行案.md`

### 执行步骤

1. **从策划案拆分 Phase**：按以下固定顺序划分

   | Phase | 内容 | 使用技能 |
   |-------|------|----------|
   | Phase 1 | 物编准备（单位/技能/装备/Buff） | `y3-obj-gen` |（强制，不可跳过）
   | Phase 2 | UI 开发（生成 JSON + 节点树，不含 Lua） | `y3-ui-pipeline` |（强制，不可跳过）
   | Phase 3 | 核心玩法（所有 Lua 逻辑 + UI 绑定） | `y3-lua-pipeline` |（强制，不可跳过）
   | Phase 4 | 扩展机制（死亡/复活/胜负等） | `y3-lua-pipeline` |（强制，不可跳过）
   | Phase 5 | lua代码review | `y3-lua-review` | （强制，不可跳过）
   | Phase 6 | 测试验收 | `y3-auto-test` |

2. **每个 Phase 必须包含**：
   - **功能清单**：模块 + 功能 + 验收点 + **参考 API**（强制）+ **参考事件**（强制）
   - **详细验收点**：每个功能对应的可验证标准
   - **参考 API 列**：标注来源（`y3/` 源码路径或 `references/` 文档位置）
   - **参考事件列**：标注监听的事件名和来源

3. **参考 API 规则（强制）**：
   > ⚠️ 每个需要 Lua 实现的功能，必须在执行案中标注参考 API 和来源。
   > 如果在 `y3/` 源码和 `y3/演示/` 中找不到对应 API，标注 `?` 并在开发前确认。
   > **禁止留空、禁止臆造。**

4. **⭐ API/事件预检（强制，新增）**：生成执行案后、开始制作前，执行预检流程（见下方详细说明）

5. **按 Phase 顺序制作**：依次激活对应子技能

6. Phase 5 | lua代码review 不可以跳过

### 🔍 API/事件预检机制（强制，执行案生成后立即执行）

> **目的**：在执行案生成后、实际编码前，提前验证所有列出的 API 和事件是否真实存在，
> 消灭"写到一半发现 API 不存在"的问题。

#### 执行案功能清单格式要求（强制）

每个 Phase 3/4 的功能清单**必须**包含以下列：

```markdown
| 模块 | 功能 | 使用 API | 使用事件 | 验收点 | API 来源 |
|------|------|---------|---------|--------|----------|
| 玩家初始化 | 创建英雄 | `y3.player.create_unit(...)` | `y3.game:event('游戏-初始化')` | 英雄出现在出生点 | `references/player.md` |
| 战斗系统 | 击杀奖励 | `player:add_gold(n)` | `y3.game:event('单位-死亡')` | 击杀后金币增加 | `references/unit.md` |
| UI 绑定 | 更新血条 | `ui:set_current_progress(v)` | `y3.game:event('单位-属性变化')` | 血条随 HP 变化 | `references/y3-ui-instance.md` |
```

#### 预检流程

```
执行案生成完成
    │
    ▼
┌─ 预检 Step 1: 提取 API 清单 ─────────────────────────────────────────┐
│ 从执行案 Phase 3/4 的功能清单中，提取所有「使用 API」和「使用事件」列    │
│ 生成待验证清单：                                                       │
│   api_list = [所有列出的 API 调用]                                     │
│   event_list = [所有列出的事件名]                                      │
└────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ 预检 Step 2: 验证 API 存在性 ────────────────────────────────────────┐
│ 对每个 API：                                                           │
│   1. 先查 references/ 目录下的参考文档                                  │
│   2. 再用 grep_search 在 y3/ 源码中搜索函数名                         │
│   3. 标注结果：                                                        │
│      ✅ 已确认 — 在 references/ 或 y3/ 源码中找到                     │
│      ❓ 待确认 — 名称相似但参数/用法需核实                             │
│      ❌ 不存在 — 找不到，可能是臆造                                    │
└────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ 预检 Step 3: 验证事件存在性 ─────────────────────────────────────────┐
│ 对每个事件名：                                                         │
│   1. 在 y3/ 源码中搜索事件字符串（如 '单位-死亡'、'游戏-初始化'）       │
│   2. 检查事件回调参数是否匹配                                          │
│   3. 标注结果：✅ 已确认 / ❓ 待确认 / ❌ 不存在                      │
└────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ 预检 Step 4: 输出预检报告 ───────────────────────────────────────────┐
│ 生成「API/事件预检报告」，追加到执行案末尾：                             │
│                                                                        │
│ ### API/事件预检报告                                                   │
│ | 项目 | 类型 | 状态 | 来源/备注 |                                    │
│ |------|------|------|-----------|                                     │
│ | y3.player.create_unit | API | ✅ | references/player.md |           │
│ | y3.game:event('单位-死亡') | 事件 | ✅ | y3/game/game_event.lua |   │
│ | unit:get_buff_count() | API | ❌ | 不存在，改用 ... | |             │
│                                                                        │
│ **预检统计**：                                                         │
│   ✅ 已确认: X 项 | ❓ 待确认: Y 项 | ❌ 不存在: Z 项               │
└────────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ 预检 Step 5: 处理问题项 ────────────────────────────────────────────┐
│ ❌ 不存在的 API/事件：                                                 │
│   - 搜索 y3/ 源码找到正确的替代 API                                   │
│   - 更新执行案中的功能清单                                             │
│   - 如果找不到替代方案，标注 ⚠️ 并 ask_user_question 协商             │
│                                                                        │
│ ❓ 待确认的 API/事件：                                                 │
│   - 查阅 y3/ 源码确认参数签名                                         │
│   - 更新执行案中的调用写法                                             │
│                                                                        │
│ 📊 预检通过条件：❌ 项 = 0 且 ❓ 项 = 0                              │
│ 未通过 → 修正后重新预检（最多 2 轮）                                   │
│ 通过 → ✅ 进入制作阶段                                                │
└────────────────────────────────────────────────────────────────────────┘
```

#### 预检参考路径

| 验证目标 | 查找路径 | 说明 |
|----------|---------|------|
| API 函数 | `skills/y3-lua-pipeline/references/*.md` | 首选：参考文档速查 |
| API 函数 | `maps/EntryMap/script/y3/` 或 `global_script/y3/` | 次选：源码搜索 |

| 常见错误 | `.codemaker/memory/lua-issues/api_issues.md` | 历史错题比对 |
| UI API | `references/y3-ui-*.md`、`references/ui-events.md` | UI 相关 API |

### 子技能路由

| 任务类型 | 路由到 | 说明 |
|----------|--------|------|
| 生成物编 | `y3-obj-gen` | 必须用脚本，禁止手写 JSON |
| 修改物编 | `y3-obj-edit` | 修改已有物编属性 |
| 生成 UI JSON | `y3-ui-pipeline` | 生成 JSON + 节点树 |

## 📋 阶段 3：转换为测试案 + 测试

### 触发条件
执行案制作完成，用户说"开始测试"或"生成测试案"。

### 输入
执行案（`{gameName}执行案.md`，fallback: `Y3AgentTestPlan.md`）

### 输出物
按 `spec-config.json` 配置输出：`{paths.testPlan}/{gameName}测试案.md`

### Gate 准入流程

> 进入测试阶段前，必须按顺序通过以下 Gate。
> ⚠️ Gate 中的文件检查需同时查找新命名和旧命名（fallback）。

```
用户说"开始测试"
    │
    ▼
┌─ Gate 1: 策划案是否存在？ ─────────────────────────────────────────┐
│ 检查: 按 spec-config.json 配置路径查找策划案/执行案文件               │
│ Fallback: openspec/docs/ 下的 GameDesign.md / Y3AgentTestPlan.md    │
│   ├─ 存在 → 进入 Gate 2                                            │
│   └─ 不存在 → ❌ 不走自动化测试，降级为普通测试                      │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Gate 2: 测试案是否存在？ ─────────────────────────────────────────┐
│ 检查: 按 spec-config.json 路径查找 {gameName}测试案.md               │
│ Fallback: openspec/docs/AutoTestPlan.md                              │
│   ├─ 存在 → 进入 Gate 2.5                                          │
│   └─ 不存在 → 按 gen-test-plan.md 规范自动生成 → 进入 Gate 2.5     │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Gate 2.5: 测试模式判断（自动检测 + 用户确认） ────────────────────┐
│ 1. 扫描工程目录，检测已有开发产出（物编/UI/Lua）                     │
│ 2. 向用户确认模式（必须使用 ask_user_question）：                    │
│   ├─ 开发+测试模式 → 进入 Gate 3（确认清理范围后执行）               │
│   ├─ 纯测试模式   → 跳过 Gate 3，直接开始验收                       │
│   └─ 指定 Phase   → 进入 Gate 3（仅确认指定 Phase 清理范围）        │
└─────────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Gate 3: Step 0 清理范围确认（⚠️ 必须用户确认） ──────────────────┐
│ ⛔ 禁止未经确认直接执行清理操作！                                     │
│ 1. 列出待清理文件清单                                                │
│ 2. 用 ask_user_question 请用户确认                                   │
│   ├─ 用户确认 → ✅ 启动自动化测试                                   │
│   └─ 用户修改 → 更新 Step 0 后再启动                                │
└─────────────────────────────────────────────────────────────────────┘
```

### 测试模式差异

| 维度 | 开发+测试模式 | 纯测试模式 |
|------|-------------|-----------|
| Step 0 清理 | ✅ 执行（需 Gate 3 确认） | ⛔ 跳过 |
| Phase 开发任务 | ✅ 执行（调用子技能） | ⛔ 跳过 |
| Phase 验收用例 | ✅ 执行 | ✅ 执行（核心） |
| Step 5 修复回归 | ✅ 执行 | ✅ 执行 |
| Step Final 报告 | ✅ 生成 | ✅ 生成 |

### 转换步骤（Gate 2 测试案不存在时）

> 📋 **详细生成规范**：`skills/y3-game-spec/gen-test-plan.md`
> 流程：Step A 解析策划案 → Step B 映射测试用例 → Step C 填充模板 → Step D 自检

1. **执行转换检查清单**（见执行案末尾的「转换检查清单」章节）：
   - 逐项检查每个验收点
   - 确认三要素齐全：**① 开发任务 + ② 参考 API + ③ 测试用例**
   - 缺失任何一项必须补齐

2. **生成测试案文档**：
   - Step 0：清理工程
   - Phase 1~4：每个 Phase 包含开发任务表 + 验收用例表
   - Step 5：问题修复 + 回归重跑
   - Step Final：生成测试报告

3. **测试案中的开发任务必须包含参考 API**：
   - 从执行案的「参考 API」列直接复制
   - 对于关键绑定 API，附上完整代码示例（如物品栏的 `set_ui_unit_slot`）

4. **生成质量自检**：12 项质量检查，不通过则自行修正后再输出

## 🔁 迭代循环

### 触发条件
测试发现问题（FAIL / PARTIAL / KNOWN ISSUE）

### 迭代规则

| 问题类型 | 回到哪个阶段 | 说明 |
|----------|-------------|------|
| **策划级问题** | 阶段 1 | 功能缺失、规则矛盾 → 修改策划案 |
| **开发级问题** | 阶段 2 | API 错误、逻辑 Bug → 修改执行案 + 代码 |
| **测试级问题** | 阶段 3 | 用例覆盖不足 → 补充测试案 |

### 迭代流程

```
测试报告 (TestReport.md)
    │
    ▼ 分析问题根因
    │
    ├─ 策划遗漏 → 修改策划案 → 更新执行案 → 更新测试案 → 重新制作+测试
    │
    ├─ 开发错误 → 修复代码 → 更新执行案（如需）→ 回归测试
    │
    └─ 用例不足 → 补充测试案 → 重跑测试
    │
    ▼ 最多 3 轮回归
    │
    输出最终 TestReport.md
```

## � Step Final：测试报告 + HTML 报告生成

### 测试报告输出

测试完成后，按 `spec-config.json` 配置生成测试报告：

| 产出 | 路径 | 说明 |
|------|------|------|
| MD 报告 | `{paths.testReport}/{gameName}测试报告-{timestamp}.md` | Markdown 格式原始报告 |
| HTML 报告 | `{paths.htmlReport}/{gameName}测试报告-{timestamp}.html` | 可视化静态 HTML 报告 |

### HTML 报告生成流程（强制）

```
测试完成，生成 MD 测试报告
    │
    ▼
运行 HTML 转换脚本：
  python scripts/gen_test_report_html.py "{md报告路径}"
    │
    ▼
脚本自动：
  1. 读取 MD 报告
  2. 解析测试结果（PASS/FAIL/PARTIAL/SKIP/KNOWN 统计）
  3. 生成带统计面板的 HTML 页面
  4. 输出到 spec-config.json 配置的 htmlReport 路径
  5. 如果 htmlReport.autoOpen = true，自动在浏览器中打开
    │
    ▼
检查 SVN 自动上传：
  如果 svn.autoUpload = true：
    1. svn add "{md报告路径}" "{html报告路径}"
    2. svn commit -m "{commitMessage}" "{md报告路径}" "{html报告路径}"
    │
    ▼
✅ 向用户展示报告路径（+ SVN 提交结果）
```

### SVN 自动上传（可选）

通过 `spec-config.json` 的 `svn.autoUpload` 开关控制。

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `svn.autoUpload` | `false` | 是否在生成测试报告后自动提交到 SVN |
| `svn.commitMessage` | `{gameName}测试报告-{timestamp}` | 提交信息，支持占位符替换 |

**前置条件**：
- 本机已安装并配置 `svn` 命令行工具
- 项目目录已是 SVN 工作副本（working copy）
- 报告输出目录在 SVN 版本控制范围内

**执行流程**：
```bash
# 1. 添加新文件到 SVN
svn add --force "{md报告路径}" "{html报告路径}"

# 2. 提交
svn commit -m "{gameName}测试报告-{timestamp}" "{md报告路径}" "{html报告路径}"
```

> ⚠️ 如果 SVN 提交失败（网络/权限等），不影响报告生成，仅输出警告信息。

### HTML 报告特性

- **顶部统计面板**：总计/通过/失败/部分/已知/跳过 六项统计卡片
- **进度条**：按比例可视化各状态占比
- **整体评级**：自动计算通过率并显示评级标签
- **表格高亮**：PASS 绿色、FAIL 红色、PARTIAL 黄色等
- **暗色主题**：护眼的深色 UI
- **纯静态**：无外部依赖，单文件即可查看
- **响应式**：支持移动端查看

### 手动生成 HTML 报告

```bash
# 基本用法
python scripts/gen_test_report_html.py openspec/docs/塔防测试报告-2026年4月14日11时12分54秒.md

# 指定输出路径
python scripts/gen_test_report_html.py report.md --output custom/path/report.html

# 不自动打开浏览器
python scripts/gen_test_report_html.py report.md --no-open
```
