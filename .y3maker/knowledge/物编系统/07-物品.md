# 物品 (Item)

## 概述

物品是可创建、拾取、使用、出售、丢弃的运行时对象，可在场景、物品栏、背包栏中流转。

**在游戏中的作用**：
- 消耗品与装备系统
- 物品技能（主动/被动）
- 属性加成与合成
- 商店买卖与库存逻辑

## 核心概念

### 物编 ID vs 运行时实例

```lua
-- 物编 ID
local item_key = 134239954

-- 运行时实例
local item = y3.item.create_item(point, item_key, player)
```

### 物品存在状态

| 状态 | 说明 | 检查方法 |
|------|------|----------|
| 场景中 | 地图地面对象 | `item:is_in_scene()` |
| 物品栏 | 单位装备栏/快捷栏 | `item:is_in_bar()` |
| 背包 | 单位背包栏 | `item:is_in_bag()` |

### 物编附加单位属性

物品物编除了能配置模型、堆叠、技能、价格以外，还可以直接配置 `attached_*` 系列字段，用来给 `持有该物品的单位` 自动增加属性。

- 这些字段是 `物品类型配置` 的一部分，不需要你在 Lua 里额外手动给单位加属性。
- 单位获得、装备、持有该物品时，这部分属性会并入单位属性体系。
- 单位失去、移除、丢弃该物品后，这部分属性也会随物品一起失效。
- 这类字段影响的是 `持有者单位属性`，不是物品实例自己的 `item:set_attr()` / `item:set_attribute()` 那组运行时属性接口。
- `attached_*` 属性统一按五段数组 `[基础值, 基础加成, 增益值, 增益加成, 总属性加成]` 来理解和配置。
- 只有五段都为 `0` 时，才可以用 `0` / `0.0` 占位；只要任意一段非 `0`，就应写成完整数组。

当前模板里能直接看到的常见字段包括：

| 字段 | 作用 |
|------|------|
| `attached_hp_max` | 附加最大生命 |
| `attached_hp_rec` | 附加生命恢复 |
| `attached_strength` | 附加力量 |
| `attached_agility` | 附加敏捷 |
| `attached_intelligence` | 附加智力 |
| `attached_critical_chance` | 附加暴击率 |
| `attached_dodge_rate` | 附加闪避率 |
| `attached_hit_rate` | 附加命中率 |
| `attached_pene_phy` | 附加物理穿透 |
| `attached_pene_mag` | 附加魔法穿透 |

示例：

```json
{
    "attached_hp_max": [100.0, 20.0, 50.0, 0.0, 10.0],
    "attached_hp_rec": [5.0, 0.0, 0.0, 0.0, 0.0],
    "attached_strength": [12.0, 0.0, 0.0, 0.0, 0.0],
    "attached_agility": [8.0, 0.0, 0.0, 0.0, 0.0],
    "attached_intelligence": 0.0,
    "attached_critical_chance": [10.0, 0.0, 0.0, 0.0, 0.0],
    "attached_dodge_rate": [0.0, 0.0, 0.0, 0.0, 5.0],
    "attached_hit_rate": [0.0, 0.0, 15.0, 0.0, 0.0],
    "attached_pene_phy": [15.0, 0.0, 0.0, 0.0, 0.0],
    "attached_pene_mag": 0.0
}
```

这表示：单位只要持有这件物品，就会自动获得这些属性加成。

如果你只想给一个固定数值，也应该把数组型字段写成 `[数值, 0, 0, 0, 0]`；不要把非零配置简写成单个数字。

需要区分两套能力：

- `attached_*`：物编层的自动加成，面向 `持有者单位`。
- `item:set_attr()` / `item:set_attribute()` / `item:set_bonus_attribute()`：Lua 运行时改的是 `物品实例上的属性数据`。

如果你要进一步研究属性字段的完整分类和计算方式，可以查看 [.codemaker/knowledge/核心系统/02-属性计算公式.md](.codemaker/knowledge/核心系统/02-属性计算公式.md)。

## 常用操作

### 位置与状态

```lua
item:get_point()

-- 仅适用于场景中的物品
item:set_point(target_point)
item:drop(point, 1)

item:is_in_scene()
item:is_in_bar()
item:is_in_bag()
```

### 堆叠与充能

```lua
item:get_stack()
item:set_stack(10)
item:add_stack(2)

item:get_charge()

item:add_charge(1)
item:get_max_charge()
item:set_max_charge(5)

item:get_stack_type() -- 0/1/2
```

### 属性与外观

```lua
item:set_attr('攻击', 20, 'ATTR_BONUS')
item:set_attribute('攻击', 50)
item:add_attribute('攻击', 5)
item:get_attribute('攻击')

item:set_bonus_attribute('攻击', 20)
item:add_bonus_attribute('攻击', 5)
item:get_bonus_attribute('攻击')

item:set_icon(icon_id)
item:get_icon()
item:set_scale(1.2)
item:get_scale()
item:set_facing(90)
item:get_facing()
item:set_visible(true)
```

### 所有者与技能

```lua
item:get_owner()
item:get_owner_player()
item:set_owner_player(player)
item:get_slot()
item:get_slot_type()

item:get_ability()
item:get_passive_ability(1)
item:add_passive_ability(ability_id, 1)
```

- `item:get_owner()`：当前持有该物品的单位。
- `item:get_owner_player()`：物品记录的创建者玩家，对应底层 `creator`。
- `item:set_owner_player(player)`：修改物品记录的创建者玩家，不等于把物品直接转移给另一个单位。

### 生命周期与标签

```lua
item:add_tag('装备')
item:has_tag('装备')
item:remove_tag('装备')

item:set_droppable(true)
item:set_sale_state(true)
item:remove()
```

## 事件监听

### 在物编对象上监听

```lua
local item_editor = y3.object.item[134239954]

item_editor:event('物品-创建', function(trg, data)
    local item = data.item
    local lua_table = data.lua_table
    print('创建物品:', item:get_name())
end)

item_editor:event('物品-获得', function(trg, data)
    local unit = data.unit
    local item = data.item
    local item_no = data.item_no
    print(unit:get_name(), '获得物品', item:get_name(), item_no)
end)

item_editor:event('物品-堆叠变化', function(trg, data)
    local item = data.item
    local delta = data.delta_cnt
    print(item:get_name(), '堆叠变化', delta)
end)
```

## 堆叠类型 (stack_type)

| 值 | 说明 |
|----|------|
| 0 | 不可堆叠；按`单个物品`处理 |
| 1 | 充能；物品本体不可堆叠，`堆叠数`固定按 1 处理，层数逻辑走 `充能数` |
| 2 | 堆叠；同类物品进入栏位时会优先并入已有堆叠，层数逻辑走 `堆叠数` |

### 堆叠类型说明

- `充能`类型的物品在运行时会被当作单个实例处理，不参与普通堆叠。
- 充能型物品的可用次数、消耗与变化事件走 `充能数` / `最大充能数`，不是 `堆叠数`。
- `堆叠`类型的物品进入物品栏或背包时，会优先并入已有同类物品。
- 并入时不是固定 `+1`，而是把来源物品当前可转移的 `堆叠数` 尽量加到目标物品上，直到目标达到 `最大堆叠数`。
- 来源物品只有在层数被完全吸收、`堆叠数 <= 0` 时才会被销毁；如果目标堆叠已满，来源物品会保留剩余层数并继续占用格子，或尝试并入其他同类物品。

### Lua 示例

下面这个场景里，如果新添加的物品被完全并入已有堆叠，`Unit:add_item()` 可能不会再返回可继续操作的物品对象；这时把返回值直接当成 `物品` 使用，就可能报错。

```lua
-- 错误示例：默认以为一定会返回一个物品对象
local ret_item = hero:add_item(item_key)
log.info(ret_item:get_name())

-- 如果新添加的这个物品被完全并入已有堆叠，ret_item 可能是 nil
-- 这里继续调用 ret_item:get_name() 就会报错
```

```lua
-- 更安全的写法：先判断返回值
local ret_item = hero:add_item(item_key)

if ret_item then
    log.info('还有剩余物品：' .. ret_item:get_name())
    log.info('剩余堆叠数：' .. ret_item:get_stack())
else
    log.info('新添加的物品已被完全并入已有堆叠，没有剩余物品返回')
end
```

## 商店配置与买卖流程

### 资源列表格式 (buy_res_list / sell_res_list)

```json
{
    "__tuple__": true,
    "items": [
        {
            "res_key": "official_res_1",
            "res_value": 100
        }
    ]
}
```

这个结构同时用于 `buy_res_list` 和 `sell_res_list`：

- `buy_res_list`：购买这个物品时，玩家需要支付的资源列表。
- `sell_res_list`：把这个物品卖给商店时，玩家会获得的资源列表。

### 商店相关字段

以下字段都能在物品元数据中找到，可作为“该物品作为商品时”的配置数据：

| 字段 | 作用 |
|------|------|
| `sale_enable` | 该物品默认是否允许出售。lualib 提供了 `item:set_sale_state(boolean)` 改写运行时出售状态，但没有封装 `item:can_sell()` 读取接口。 |
| `buy_res_list` | 购买该物品时需要的资源列表。 |
| `sell_res_list` | 出售该物品时返还的资源列表。 |
| `init_stock` | 该物品作为商品时的初始库存。 |
| `max_stock` | 商店商品库存上限。 |
| `start_rft` | 该物品作为商品时的开始售卖时间。 |
| `refresh_interval` | 该物品作为商品时的库存恢复间隔，每次恢复1点库存 |

需要注意：这些字段定义的是 `这个物品作为商品时的价格/库存元数据`，并不等于“它一定会出现在某个商店里”。

- 物品是否真的在某个商店出售，取决于 `商店单位` 的页签商品配置。
- 也就是说，`buy_res_list` / `sell_res_list` / 库存字段定义的是 `怎么卖`，不是 `在哪卖`。

### 商店购买与出售流程

就 y3 lualib 当前直接提供的接口来说，可以可靠使用的是以下能力：

- `y3.item.check_precondition_by_key(player, item_key)`：检查物品类型前置条件。
- `y3.item.get_item_buy_price_by_key(item_key, key)`：读取某项玩家资源下的购买价格。
- `y3.item.get_item_sell_price_by_key(item_key, key)`：读取某项玩家资源下的出售价格。
- `item:set_sale_state(boolean)`：改写实例当前的可出售状态。
- 事件侧可以监听 `物品-出售`、`单位-购买物品`、`单位-出售物品` 等事件。

示例：

```lua
local item_key = 134239954

if y3.item.check_precondition_by_key(player, item_key) then
    local buy_gold = y3.item.get_item_buy_price_by_key(item_key, 'gold')
    local sell_gold = y3.item.get_item_sell_price_by_key(item_key, 'gold')
    log.info('buy gold = ' .. buy_gold)
    log.info('sell gold = ' .. sell_gold)
end
```

如果你关心的是引擎侧常见的底层执行链路，可以继续参考下面这两段流程说明；它们用于描述购买/出售时通常会发生什么，不等于 lualib 已经提供了对应的高层 Lua 封装。

#### 普通购买流程

1. 购买方先发起“向某个商店购买某个页签中的物品”。
2. 源码先检查商店是否有效、购买者是否在商店范围内、是否有阵营/玩家身份、背包功能是否开启。
3. 如果买的是物品，还会额外检查购买者是不是幻象，以及当前背包是否还能接收这个物品。
4. 商店侧继续检查：目标页签里是否真的有这件商品、当前库存是否足够、购买者资源是否足够。
5. 检查通过后，商店先扣购买者资源，再扣商店库存。
6. 然后底层创建运行时物品对象，并发送购买相关事件。
7. 新建出来的物品会再走一次普通 `add_item()` 流程：优先并入已有堆叠，再尝试放入物品栏/背包。
8. 如果物品带自动使用逻辑，还会走自动使用分支；如果最终放不下，剩余物品会掉到单位脚下场景中。

#### 出售流程

1. 出售时先检查商店有效、出售者在范围内、物品确实属于这个单位。
2. 然后检查该物品当前是否允许出售；默认读物编字段 `sale_enable`，也可能被运行时脚本改写。
3. 检查通过后，会先发送出售相关事件，再把物品从单位身上移除。
4. 物品移除成功后，商店按照 `sell_res_list` 把对应资源返还给出售者。

对于“商店内部完整购买、扣库存、补货”的执行细节，当前这份 y3 lualib 里没有单独的高层 Lua 封装可直接引用，因此这里不再把这些底层流程当作已由 lualib 明确暴露的行为来描述。

## 物品合成配置、流程与时序

### 合成配方格式 (compose_list)

```json
{
    "__tuple__": true,
    "items": [
        {
            "item_key": 100001,
            "item_num": 2
        }
    ]
}
```

- `__tuple__ = true`：表示这是编辑器导出的元组结构。
- `items`：配方条目列表，每一项代表一种原料。
- `item_key`：原料物品的物编 ID。
- `item_num`：这条原料需要的数量。

### lualib 对合成的支持范围

y3 lualib 中与合成直接相关、能明确查到的内容主要有三类：

- 物编字段：`compose_list` 作为物品类型字段存在。
- 事件元数据：存在 `单位-物品合成`、`单位-购买物品合成` 事件类型。
- Lua 工具与演示：提供了 `y3.tools.synthesis` 通用合成工具，以及基于它写的演示代码。

当前这份 lualib 里没有提供“自动读取物品 `compose_list` 并在运行时自动完成合成”的高层封装接口

如果你想在纯 Lua 里实现合成，推荐直接用 `y3.tools.synthesis`：

```lua
require 'y3.tools.synthesis'

local maker = New 'Synthesis'()
maker:register(134245520, {201390001, 201390002})

y3.game:event('物品-获得', function(trg, data)
    local item_keys = {}
    for _, item in ipairs(data.unit:get_all_items():pick()) do
        item_keys[#item_keys + 1] = item:get_key()
    end

    local res = maker:check(item_keys)
    if not res then
        return
    end

    for _, material_key in ipairs(res.lost) do
        data.unit:remove_item(material_key, 1)
    end

    data.unit:add_item(res.get)
end)
```

上面这种写法和 y3 演示里提供的思路一致：

- 用 `maker:register(result, ingredients)` 手工注册配方。
- 在 `物品-获得` 之类的事件里自己决定何时检查合成。
- 用 `unit:remove_item()` 扣材料，再用 `unit:add_item()` 添加产物。

### 合成相关事件

y3 事件元数据里存在以下合成事件：

| 事件名 | 主要字段 |
|------|------|
| `单位-物品合成` | `unit`, `compose_id`, `item_prop` |
| `单位-购买物品合成` | `unit`, `shop_unit`, `item` |

但这些事件的存在，只能说明引擎侧有对应事件元数据；不能直接推出“lualib 已经把 `compose_list` 自动接成完整运行时系统”。如果你要在 Lua 层做可控的合成逻辑，仍应显式写自己的注册与检查代码。

如果你关心的是引擎侧常见的底层执行链路，可以继续参考下面这几段流程说明；它们用于帮助理解自动合成和购买合成的常见时序，不等于 lualib 已经提供了对应的高层 Lua 封装。

### 自动合成逻辑

编辑器侧常见的自动合成理解方式，一般是 `物品进入单位的物品体系后` 再做检查，不是简单地在 `add_item()` 内部直接同步完成。

完整链路通常可概括为：

1. 初始化时，会读取所有合成配方，建立 `原料物品 -> 可合成物品` 的索引。
2. 单位的物品处理器初始化后，会注册一个专门用于合成判断的内部流程点；旧版资料里常记作 `UNIT_ADD_ITEM_FOR_COMPOSE`，但它不是当前可直接在 y3 中使用的事件名。
3. 当物品进入 `物品栏` 或 `背包` 后，正常的加物品事件会先依次触发。
4. 上述事件发完后，再开始检查自动合成。
5. 自动合成检查会根据 `本次新增物品的物编 ID`，查出所有“把它当作原料”的配方。
6. 对每个配方，会循环判断 `资源是否足够` 与 `原料物品是否足够`；只要仍然满足，就会持续执行合成，不是只合成一次。
7. 合成时会先扣资源，再扣原料物品，然后创建合成产物。
8. 合成产物创建后，会先抛出一次 `合成事件`，再把产物重新走一次 `add_item()`。
9. 因为合成产物会再次进入 `add_item()`，所以如果产物本身又是别的配方原料，就会继续触发下一轮自动合成，形成连锁合成。

### 自动合成的触发条件

- 稳定会触发自动合成的入口通常是 `单位获得物品`，例如：拾取物品、脚本加物品、合成产物重新入包。
- 自动合成一般不是“资源变化就自动补触发”。
- 这意味着：如果原料已经齐了，但当时资源不够，没有发生合成；之后玩家单纯获得资源，通常还需要再发生一次“加物品”相关流程，才会重新进入这套检查。

### 合成产物的落点规则

合成产物创建出来后，通常不会直接固定进入某个栏位，而是复用普通的 `add_item()` 流程，因此落点规则和普通获得物品基本一致：

1. 先按单位当前的默认拾取槽位类型尝试放入。
2. 如果默认目标是 `物品栏`，则先尝试进 `物品栏`，失败后再尝试 `背包`。
3. 如果默认目标是 `背包`，则先尝试进 `背包`，失败后再尝试 `物品栏`。
4. 如果目标栏位里已有同类可堆叠物品，会先尝试并入已有堆叠。
5. 如果产物是自动使用物品，且正常栏位放不下，会进入备用的自动使用分支，再继续后续逻辑。
6. 如果最终仍然放不下，产物可能会掉到单位脚下场景中。

### 与商店合成购买的区别

- 普通自动合成：由 `获得物品后` 的自动检查触发。
- 商店合成购买：走单独的购买合成逻辑，会先计算“背包已有原料 + 商店补足原料 + 资源消耗”，然后再创建目标物品。

#### 商店合成购买流程

1. 商店合成购买不是简单“直接买成品”，而是先读取目标物品的 `compose_list`。
2. 会先统计购买者自己背包里已有多少原料，再计算还差多少原料需要由商店库存补足。
3. 最终资源消耗 = `配方自身的资源消耗` + `需要补买的原料价格`。
4. 如果原料库存、玩家资源、背包空间都满足，就会先扣资源，再扣掉背包里的已有原料，同时扣掉商店里补买原料的库存。
5. 之后才创建目标物品，并继续走普通 `add_item()` 流程。

### 时序注意

- 按常见的底层时序理解，自动合成检查发生在 `物品-进入物品栏` / `物品-进入背包` 等加物品事件之后。
- `合成事件` 的抛出时机通常在“合成产物创建完成之后、重新进入背包/物品栏放入流程之前”。
- 因此不要假设你在 `合成事件` 里拿到的产物，已经稳定落到某个格子里。

## 注意事项

1. 事件中的堆叠/充能变化字段是 `delta_cnt`，表示变化量。
2. 运行时对象移除后应先判断 `item:is_exist()` 再继续调用。
3. `unit:add_item()` 的参数是物编 `item_key`，不是运行时 `item_id`；返回值也可能是 `nil`。
4. `item:get_owner_player()` / `item:set_owner_player()` 操作的是物品记录的 `creator` 玩家；当前持有单位应通过 `item:get_owner()` 获取。
5. `item:set_point()` 在封装层会直接传送物品；若物品当前在单位身上，实际表现取决于引擎侧，不建议把它当作“背包内移动接口”使用。
6. 物品属性推荐用 `set_attr` / `set_attribute` / `set_bonus_attribute` 这组接口管理。
7. 如果要在 Lua 层实现自动合成，建议显式使用 `y3.tools.synthesis` 和物品事件，不要假设 `compose_list` 会被 lualib 自动执行。
