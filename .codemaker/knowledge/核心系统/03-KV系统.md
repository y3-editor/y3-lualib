# 自定义键值系统 (KV System)

## 概述

自定义键值存储在物编 JSON 文件的 `"kv"` 字段中，支持多种数据类型和引用类型。

**文件位置**：物编 JSON 文件（如 `maps\EntryMap\editor_table\editorunit\<物编ID>.json`）

## KV 数据结构

```json
{
    "kv": {
        "attr_name": {
            "key": "attr_name",        // 键名（唯一标识）
            "prop_cls": "PText",       // 属性类（存储类型）
            "type": 0,                 // 数据类型编号
            "etype": 0,                // 扩展类型编号（决定编辑器UI）
            "value": "",               // 默认值
            "desc": "",                // 描述
            "annotation": "",          // 注解
            "remark": "",              // 备注
            "show_in_attr": true,      // 是否在属性面板显示
            "sort": 0                  // 排序顺序
        }
    }
}
```

## prop_cls 存储类型

| prop_cls | 说明 | 适用场景 |
|----------|------|----------|
| `PText` | 文本存储 | 字符串 |
| `PFloat` | 浮点数存储 | 实数 |
| `PInt` | 整数存储 | 整数、引用类型（ID） |

## 使用示例

### 在物编中定义自定义KV

```json
{
  "kv": {
    "damage_bonus": {
      "key": "damage_bonus",
      "prop_cls": "PFloat",
      "value": 1.5
    },
    "target_unit": {
      "key": "target_unit",
      "prop_cls": "PInt",
      "etype": 7,
      "value": 134217728
    }
  }
}
```

### 在Lua中读取KV值

```lua
-- 获取单位物编的KV值
local damage_bonus = unit:kv_load('damage_bonus', 'number')

-- 获取技能物编的KV值
local target_range = ability:kv_load('target_range', 'number')

-- 获取字符串KV
local buff_name = modifier:kv_load('custom_name', 'string')
```

## 纯Lua开发：更常用 Storage 类

在偏 Lua 代码驱动的项目中，运行时状态通常更推荐放在 `Storage`，而不是都写成物编 KV。

### Storage 的定位

- KV：更适合「配置数据」，由编辑器维护，随物编定义。
- Storage：更适合「运行时状态」，由 Lua 逻辑维护，读写轻量、无需改物编。

### Storage 常用API

`Storage` 核心接口在 `y3.util.storage` 中：

- `obj:storage_set(key, value)`：写入任意 Lua 值。
- `obj:storage_get(key)`：读取值，不存在时返回 `nil`。
- `obj:storage_all()`：拿到底层 table（可用于批量操作）。

常见对象（如 `Unit`、`Ability`、`Item`、`Buff`、`Projectile`、`Player`、`Cast`、`Mover`）都扩展了 `Storage`，可直接调用。

### 典型用法（运行时缓存/状态）

```lua
-- 1) 从物编KV读取一个配置值（KV 本身可读可写）
local crit_rate = unit:kv_load('crit_rate', 'number') or 0

-- 2) 将运行时状态放到Storage（也可按需回写到KV）
unit:storage_set('battle.crit_rate', crit_rate)
unit:storage_set('battle.combo', 0)

-- 3) 战斗中持续读写Storage
local combo = unit:storage_get('battle.combo') or 0
unit:storage_set('battle.combo', combo + 1)

-- 4) 对玩家做会话级缓存
local kill_count = player:storage_get('stat.kill_count') or 0
player:storage_set('stat.kill_count', kill_count + 1)
```

### 选型建议

- 纯 Lua 项目，且不需要在 ECA 侧读取/设置对象键值：优先 `Storage`。
- 需要策划可视化配置、可复用模板，或需要 Lua/ECA 双侧互通：优先 `KV`（KV 天然支持两侧读写）。
- 需要运行期高频、临时、复杂状态：优先 `Storage`。
- 不建议在同一批业务状态上混用 `KV` 与 `Storage`；按场景二选一，统一维护。

### 注意事项

- `Storage` 是 Lua 侧运行时数据容器，不等同于存档系统。
- 基于当前 `y3.game.kv` 实现，`kv_save` 的可写类型是白名单：`boolean / integer / number / string / table`，以及少量对象类型（`Unit`、`Ability`、`Item`、`Buff`、`Point`、`Player`、`Projectile`、`Destructible`、`Particle`、`Mover`、`UnitGroup`、`UIPrefab`、`SceneUI`）。
- `kv_load` 的读取类型范围通常比 `kv_save` 更大（可传入更多 KV 类型枚举）；因此“能读不一定能写”是正常现象。
- 如果要写入白名单之外的类型，需要直接调用底层 `GameAPI.set_*_kv` 接口，或改为在 KV 中存基础标识（如 key/id），运行时再转换。
- 实践上建议先确定单一主方案：要么全程使用 `KV`（侧重 Lua/ECA 互通），要么全程使用 `Storage`（侧重纯 Lua 运行时状态）。
- 如果对象被对象池复用，历史 `storage` 数据可能被带上；必要时请在对象初始化时主动清理或覆盖关键字段。

### 运行时写入可行性矩阵（常见类型）

| 类型示例 | `kv_save` 直写 | 备注 |
|---------|----------------|------|
| `boolean / integer / number / string / table` | ✅ 支持 | 来自 `KV.SupportType` 白名单。 |
| `Unit / Ability / Item / Buff / Point / Player` | ✅ 支持 | 对象会经过 `py_converter` 转换后写入。 |
| `Projectile / Destructible / Particle / Mover / UnitGroup / UIPrefab / SceneUI` | ✅ 支持 | 同上，属于白名单对象类型。 |
| `Vector3 / Rotation / PointList / Rectangle / RoundArea / Polygon` | ⚠️ 通常不支持直写 | `kv_load` 常可读；写入需看是否有对应 `GameAPI.set_*_kv`。 |
| `Model / Texture / SfxKey / AudioKey / Sequence / CursorKey` | ⚠️ 通常不支持直写 | 多数场景建议存资源 key（字符串/整数）或走底层接口。 |
| `PlayerGroup / Team / Camera / Light / Fog / Physics*` | ⚠️ 通常不支持直写 | 偏枚举/复杂引用，优先用底层接口或改存基础标识。 |
| 运行时临时对象、闭包、函数、自定义复杂 table 嵌套 | ❌ 不建议写 KV | 推荐放 `Storage`；KV 更适合配置和可互通数据。 |

补充说明：

- 对“物编对象KV”（`unit_key/item_key/ability_key/...`）可用的 `set_*_kv` 接口覆盖面，可能与“运行时对象 handle 的 add_*_kv”不同。
- 因此同一种值类型，可能出现“某类对象可写、另一类对象不可写”的情况；遇到时报错 `不支持的类型` 时，应回看当前对象的写入路径。
